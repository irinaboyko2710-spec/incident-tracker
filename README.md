# Трекер інцидентів кібербезпеки (Лабораторна робота №5)

## Опис проєкту
У межах цієї лабораторної роботи проведено комплексний аудит безпеки (Security Hardening) бекенд-додатка на Express. Робота була спрямована на виявлення та усунення критичних вразливостей (SQLi, XSS, IDOR, Misconfiguration) та переведення системи на стандарти безпечної розробки.


## Детальний опис виконаних заходів (Сценарії А-Г)

### Сценарій А: Захист від SQL Injection (SQLi)
* **Проблема:** Пошук та фільтрація даних у репозиторіях базувалися на конкатенації рядків, що дозволяло зловмисникам маніпулювати SQL-запитами.
* **Рішення:** Проведено рефакторинг запитів до бази даних SQLite. Впроваджено використання **параметризованих запитів** із плейсхолдерами (`?`). Це гарантує, що введені користувачем дані обробляються виключно як текстові літерали, унеможливлюючи виконання стороннього SQL-коду.

### Сценарій Б: Захист від Stored XSS
* **Проблема:** Фронтенд-частина програми використовувала небезпечну властивість `innerHTML` для динамічного виводу даних із бази. Це створювало умови для виконання довільного JavaScript-коду, введеного через поля форми.
* **Рішення:** Повністю відмовлено від `innerHTML`. Оновлено логіку рендерингу елементів: тепер дані додаються до DOM виключно через **`textContent`**. Це забезпечує автоматичне екранування будь-яких HTML-тегів або скриптів, відображаючи їх як безпечний текст.

### Сценарій В: Захист від IDOR (Broken Access Control)
* **Проблема:** Доступ до записів інцидентів здійснювався виключно за їхнім унікальним ID в URL, без перевірки того, чи належить цей запис конкретному авторизованому користувачу.
* **Рішення:** Логіку перевірки прав доступу перенесено безпосередньо на бекенд. До кожного SQL-запиту (GET, PUT, DELETE) додано умову перевірки `owner_id`. Якщо ID інциденту не збігається з ID поточного користувача, сервер повертає статус **`403 Forbidden`**.

### Сценарій Г: Security Misconfiguration (Мінімальний харднінг)
* **Безпечні заголовки:** Додано мідлвару для встановлення HTTP-заголовків: `X-Frame-Options: DENY` (захист від Clickjacking), `X-Content-Type-Options: nosniff` (блокування MIME-sniffing), `X-XSS-Protection: 1; mode=block`.
* **Захист від перебору:** Інтегровано `express-rate-limit`, що обмежує кількість запитів з однієї IP-адреси, запобігаючи Brute-force атакам.
* **Безпечна обробка помилок:** Розроблено централізований `errorHandler`. Він приховує технічні деталі інфраструктури (стек-трейси, шлях до БД) і трансформує всі системні збої у стандартизовану JSON-відповідь.
* **Суворий CORS:** Замість відкритого доступу (`*`) налаштовано політику доступу лише для довіреного фронтенду (`http://127.0.0.1:5500`).
